This document describes an API to discover and communicate with devices over the Bluetooth 4 wireless standard using the Generic Attribute Profile (GATT).

Changes to this document may be tracked at https://github.com/WebBluetoothCG/web-bluetooth/commits/gh-pages.

Introduction

Bluetooth is a standard for short-range wireless communication between devices. Bluetooth "Classic" (BR/EDR) defines a set of binary protocols and supports speeds up to about 24Mbps. Bluetooth 4.0 introduced a new "Low Energy" mode known as "Smart" or BTLE which is limited to about 1Mbps but allows devices to leave their transmitters off most of the time, and provides most of its functionality through a single key/value protocol known as the Generic Attribute Profile (GATT).

GATT allows device makers to advertise that they support particular Services. Each Service is identified by a UUID ([[!RFC4122]]) and exposes a collection of included Services and Characteristics. Each Characteristic is identified by a UUID exposes a typed value, some properties to describe how the value can be used, and a collection of Descriptors. Each Descriptor is identified by a UUID and contains related information about the Characteristic Value.

BTLE also defines multiple roles that devices can play. The Broadcaster and Observer roles are for transmitter- and receiver-only applications, respectively. The Peripheral role is able to receive a single connection. The Central role supports multiple connections, and is responsible for creating any connections to Peripheral devices. GATT further defines Client and Server roles, which are orthogonal to the Peripheral and Central roles.

The first version of this specification focuses on the Bluetooth 4 GATT protocol in the Central and Client roles. While this specification cites the [[BLUETOOTH41]] specification, it intends to also support communication with devices that only implement Bluetooth 4.0.

Security and privacy considerations

Device Discovery

Promise<BluetoothDevice> requestDevice(sequence<BluetoothScanFilter> filters)
When this method is invoked, the UA MUST run the following steps:
  1. Return a new Promise, but continue running these steps asynchronously.
  2. If the current document's origin is an insecure origin, reject the Promise with a SecurityError and abort these steps.

    Follow slightlyoff/ServiceWorker#385 for the right term to use here.

  3. If the algorithm is not allowed to show a popup, reject the Promise with a SecurityError and abort these steps.
  4. Perform a scan for devices matching filters.
  5. Display a prompt to the user requesting that the user specify some devices from the result of the scan. The UA SHOULD show the user the human-readable name of each device. If this name is not available because the UA's Bluetooth system doesn't support privacy-enabled scans, the UA SHOULD allow the user to indicate interest and then perform a privacy-disabled scan to retrieve the name.

    The UA MAY allow the user to select a nearby device that does not match filters.

  6. Wait for the user to have made their selection.
  7. If the user cancels the prompt, reject the Promise with a NotFoundError and abort these steps.
  8. Connect to the device. ([[BLUETOOTH41]] 3.G.6.2.1) If the connection fails, reject the Promise with a NetworkError and abort these steps.
  9. Resolve the Promise with a BluetoothDevice instance representing the selected device.
sequence<BluetoothServiceId> services
A list of Service UUIDs that a device must support to match this filter.

A device matches a filter filter if

  • filter.services is not empty and
  • the device supports each of the Service UUIDs included in filter.services as a primary (vs included) service.

To perform a scan for devices matching a sequence of BluetoothScanFilters, the UA MUST put its Bluetooth subsystem into either passive scanning or privacy-enabled active scanning mode and wait a UA-determined time for nearby devices to advertise themselves. The result of the scan is the sequence of devices that match any filter in the sequence of BluetoothScanFilters. When determining the list of Service UUIDs a device supports, the UA MUST include Services advertised by the device, and MAY include Services previously discovered through the Bluetooth Service Discovery Protocol (SDP).

UUID registeredUUID(unsigned long alias)
Returns a 128-bit UUID given a 16- or 32-bit Bluetooth UUID alias. The algorithm for converting an alias to a full 128-bit UUID is defined in [[!BLUETOOTH41]] Volume 3 Part B Section 2.5.1: the top 32 bits of "00000000-0000-1000-8000-00805F9B34FB" are replaced by the bits of the alias. For example, registeredUUID(0xDEADBEEF) returns UUID.parse("DEADBEEF-0000-1000-8000-00805F9B34FB").
